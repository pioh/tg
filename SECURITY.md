# Безопасность

`tg` запускает ИИ-агента в твоём **реальном** Telegram-аккаунте. Это **личный локальный**
инструмент: агент работает с **полными правами** на твоей машине (как Claude Code без
ограничений — все инструменты, включая Bash), потому что управляешь им только ты.
Запускай его **только на своей машине и со своими данными**.

Модель осознанно простая: **отправка в Telegram технически свободна** (никакого
код-гейта «кому можно писать» — он только мешал, а агент-бот всё равно мог его снять).
Дисциплина — в правилах: по умолчанию агент **пассивен** и пишет посторонним только по
твоей явной просьбе/правилу. Кодом запрещена лишь утечка **чувствительных файлов**.

## Модель угроз (коротко)

| Угроза | Защита |
| --- | --- |
| Слив секретов файлом (сессия, токен бота, `.env`, конфиги тенантов) | `send_file` блокирует их по `realpath` у ВСЕХ тенантов и legacy `data/`; `tenants/`/`data/` в `.gitignore`; секреты маскируются в логах. |
| Агент пишет не тому в Telegram | Дисциплина в `rules/50` (пассивность по умолчанию). Жёсткого код-гейта нет — это личный инструмент, которым управляешь только ты. |
| Кто может писать боту | По умолчанию только владелец; остальные — только из allowlist (`/allowuser`). |
| Path traversal через имя тенанта | Имя валидируется (`tenantDir`/`isTenantName`): только `[A-Za-z0-9_-]`, без `..`/слэшей. |
| Локальный процесс дёргает хаб | RPC-хаб на 127.0.0.1 требует bearer-токен из `service.lock` тенанта (права 600). |
| Два сервиса портят сессию | Lock-файл на тенанта: второй экземпляр не стартует. |
| Утечка при публикации репозитория | `bun run tg doctor --prepublish` проверяет, что личные данные не в git. |

> ⚠️ Полные локальные права означают, что агент (и потенциально prompt-injection) может
> выполнять любые локальные действия — читать файлы, запускать команды, писать в любой чат.
> Это осознанный выбор для ЛИЧНОГО инструмента; не запускай его на машине с чужими
> секретами или в недоверенной среде.

## Что разрешено по умолчанию

- **Чтение** Telegram — свободно (статус «прочитано» при чтении истории не меняется).
- **Отправка** — технически в любой чат, но по правилам агент пассивен: пишет посторонним
  только когда ты явно попросил или завёл правило (например, монитор `action=reply`).
- **Отметка прочитанным** (`tg_mark_read`) — запрещена без явного подтверждения
  (`confirm=true`), т.к. тебе важно видеть непрочитанные.
- **Отправка файлов** — `send_file` не отправит чувствительные файлы (сессия, `config.json`,
  `state.json`, `service.lock`, `permissions.json`, `.env`) ни у одного тенанта.

## Аутентификация движка

- **Claude**: без `ANTHROPIC_API_KEY` используется локальная авторизация Claude Code
  (`claude setup-token`). С ключом — оплата через API. `bun run doctor` показывает режим.
- **Codex**: `codex login` (подписка ChatGPT) или `OPENAI_API_KEY`.

Агент работает с полными правами: Claude — `permissionMode: "bypassPermissions"` (все
инструменты, включая Bash), Codex — `sandboxMode: "danger-full-access"`, `approvalPolicy:
"never"`. Это сознательно: tg — личный локальный инструмент. Единственное, что ограничено
кодом, — отправка чувствительных файлов (`send_file` → `assertSafeFile`).

## Чувствительные файлы (никогда не коммить и не отправлять)

Для каждого пользователя — в его папке `tenants/<имя>/` (или legacy `data/`):

- `session/*` — сессия Telegram (полный доступ к аккаунту!)
- `config.json` — `api_id/api_hash`, токен бота
- `permissions.json`, `service.lock`
- `.env`

Папки `tenants/` и `data/` целиком в `.gitignore`. `api_id/api_hash` по умолчанию встроены (общие, как в исходной
заготовке) — это сделано осознанно, чтобы не регистрировать своё приложение; хочешь свои
— задай `TG_API_ID`/`TG_API_HASH`.
