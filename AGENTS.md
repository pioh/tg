# tg — ИИ-агент личного Telegram (точка входа для Codex)

Содержательные инструкции и память агента живут в общей иерархии (одинаковой для
Codex и Claude Code), а не в этом файле.

## Прочитай первым

1. `rules/00-agent-core.md` — кто ты и доктрина «память на диске».
2. `rules/10-memory-hierarchy.md` — структура памяти и как её вести.
3. Выбери пользователя: MCP-инструмент `list_tenants`, затем `set_context(<имя>)`. Без
   этого остальные MCP-инструменты (включая `mem_bootstrap`) вернут ошибку (намеренно —
   никаких скрытых дефолтов/авто-выбора тенанта).
4. Загрузи память: вызови MCP-инструмент `mem_bootstrap` (правила + `data/handoff.md`
   + свежие `data/qa` + хвост `data/progress.txt`).
5. Остальное: `rules/20-telegram-tools.md`, `rules/30-chat-handling.md`,
   `rules/40-reply-style.md`, `rules/50-safety.md`.

## Где что менять

- **Мультитенант**: у каждого пользователя своя рабочая папка `tenants/<имя>/` (до
  миграции — общая `data/`). «`data/`» ниже = рабочая папка ТЕКУЩЕГО пользователя;
  `mem_*`-инструменты сами пишут в неё.
- Память и личные правила меняй **только в рабочей папке** (через `mem_*`-инструменты
  или прямым редактированием) — НИКОГДА в base `rules/` и не в этом файле. Так изменения
  увидит любой следующий агент любой платформы.
- Просьбы человека фиксируй дословно в `data/qa`; правила храни в финальном виде;
  актуальный статус — в `data/handoff.md`.

## MCP

Инструменты Telegram и памяти даёт MCP-сервер `telegram` из `.codex/config.toml`.
Это ТОНКИЙ ПРОКСИ к запущенному сервису: с Telegram работает только сервис
(единственный владелец сессии). Если инструменты отвечают «Сервис не запущен» —
запусти `bun run service` (один экземпляр).

Codex подхватывает проектный конфиг только для **доверенного** проекта — подтверди
доверие при первом запуске (или добавь проект в `~/.codex/config.toml`,
`[projects."<abs-путь>"] trust_level = "trusted"`).
