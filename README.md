# tg — ИИ-агент для личного Telegram

Подключает ИИ-агента (**Claude Code** или **Codex**) к твоему **личному** Telegram.
Работает на [Bun], память хранит на диске, по умолчанию **пассивен** и ничего не
делает в чатах без твоих явных правил.

> ⚠️ Агент действует в твоём **реальном** аккаунте от твоего имени. Это мощно и
> потенциально рискованно — прочитай раздел [Безопасность](#безопасность) и
> [SECURITY.md](SECURITY.md). Отправка кому-либо, кроме тебя самого, разрешена
> **только** там, где ты явно разрешил (это проверяется в коде, а не только в промпте).

## Быстрый старт

```bash
bun install
bun run setup        # спросит имя (напр. me), вход в Telegram (QR), создаст бота — ведёт агент
bun run service      # запустить агента (поднимает всех настроенных пользователей)
```

`setup` сам поднимает сервис и проводит через настройку — README читать не обязательно.
Нужны установленные `bun` и движок (`claude` или `codex`) в `PATH`.

**Несколько пользователей в одном сервисе.** Каждый пользователь — это отдельная папка
`tenants/<имя>` со своей сессией, ботом и памятью. Добавить ещё одного: `bun run setup
<имя>`. `bun run service` поднимает их всех сразу. Список: `bun run tg tenant list`.

## Что умеет (честно)

- **Видит контекст сообщений**: текст, ответы/пересылки, группы, каналы, **топики**
  форумов; типы и метаданные медиа (фото, видео, голосовые, документы, опросы, гео).
- **Смотрит картинки**: фото/изображения отдаёт модели инлайн (vision) — реально *видит*.
- **Голосовые/видео**: скачивает и видит метаданные (длительность и т.п.).
  Транскрипции пока **нет** (TODO).
- **Пассивен по умолчанию**: без твоих правил ничего в чатах не делает и не трогает
  шумные группы.
- **Мониторы**: следить за конкретным человеком / чатом / топиком. Триггеры: «не чаще
  раза в N сек», «только если ты сам не ответил за N сек». Действие: уведомить /
  черновик / ответить.
- **Отвечает от твоего имени — только с разрешения**. Авто-ответы требуют либо монитора
  `action=reply`, либо явного `permission_grant`. Запрет на отправку посторонним
  **встроен в код**.
- **Сервисный бот**: отдельный бот, через который ты переписываешься с агентом и даёшь
  команды; бот пишет тебе сам (например, **сводка каждые 5 минут**). Подсказки команд
  (`/help`, `/start`), индикатор **«печатает…»** пока агент работает, ответы в формате
  **MarkdownV2**, длинные — автоматически бьёт на части.
- **Не отмечает прочитанным** за тебя (требует явного подтверждения).
- **Память на диске**: правила, заметки, дословные просьбы, статус — в папке пользователя
  `tenants/<имя>/`; переживает перезапуски, общая для Claude Code и Codex.
- **Прозрачный аудит**: каждое действие — в консоли сервиса и `data/actions.log`
  (секреты маскируются).

## Реакция: насколько «мгновенно»

- **Бот** — near-instant: long-poll `getUpdates`, реагирует сразу.
- **Личные чаты / мониторы / расписания** — лёгкий **детектор каждые ~3с** (polling,
  не push). Тяжёлый ход агента стартует только когда реально есть событие.

## Как управлять

Пиши команды агенту в **«Избранное»** или **сервисному боту** обычным текстом:
- «следи за чатом с мамой, напомни если не отвечу 15 минут»
- «каждые 5 минут присылай сводку новых сообщений в бота»
- «отвечай Алексею короткой цитатой на каждое его сообщение»

Агент запишет просьбу дословно (`data/qa`), заведёт монитор/расписание/правило и дальше
работает сам. Его текстовые ответы приходят прямо в чат бота (формат **MarkdownV2**,
длинное режется само), а пока он работает — виден индикатор **«печатает…»**, чтобы было
понятно, что он жив и занят.

**Команды боту** (детерминированные, без LLM; видны в меню `/` Telegram):
- `/help`, `/start` — справка и возможности;
- `/context` — контекст и расход сессии; `/new` — новая сессия; `/compact` — сжать контекст;
- `/model <m>`, `/effort <low|medium|high|xhigh|max>` (для Codex — `minimal…xhigh`, без `max`);
- `/monitors`, `/schedules`, `/permissions` — что настроено;
- `/grant <chat>` / `/revoke <chat>` — разрешить/отозвать ответы агента в чат;
- `/users`, `/allowuser <id|@user>`, `/denyuser <id|@user>` — кто может писать боту;
- `/here`, `/forgethere` — (в группе) сделать её общей семейной / отключить;
- `/version`, `/update`, `/restart` — версия/обновление/перезапуск (см. «Обновление»).

**Кто может писать боту.** По умолчанию — **только ты** (аккаунт, под которым вошёл в
Telegram). Если боту напишет кто-то ещё, бот сообщит тебе об этом и предложит `/allowuser
<id>`. Добавленные пользователи могут общаться с агентом, но команды управления — только
для владельца.

**Семья: общий бот в группе.** Добавь бота в общую группу и отправь там `/here` — бот
станет отвечать на сообщения всех разрешённых участников прямо в группе (несколько людей
общаются с одним ботом). Чтобы он видел обычные сообщения, выключи у бота privacy-mode
через @BotFather (Bot Settings → Group Privacy → Turn off). Кто допущен — управляется
теми же `/allowuser` / `/denyuser`. Отключить группу — `/forgethere`.

## Фоновый сервис (работает постоянно)

Чтобы агент работал всегда и сам поднимался после перезагрузки — поставь его как
**пользовательский сервис** (нативно для ОС). На первой `bun run setup` мастер сам это
предложит; можно и вручную:

```bash
bun run tg install-service     # Linux → systemd --user, macOS → launchd (Windows — инструкция)
bun run tg service-help        # где логи, как остановить/запустить/перезапустить
bun run tg uninstall-service   # удалить фоновый сервис
```

Например, на Linux логи смотрят так: `journalctl --user -u tg-agent -f`, остановка —
`systemctl --user stop tg-agent`. На macOS логи — в `data/service.log`.

## Обновление

Версионирование — [SemVer](https://semver.org), все изменения — в
[CHANGELOG.md](CHANGELOG.md). Обновиться можно тремя способами:

- **В боте**: `/version` — проверить, `/update` — обновить (агент сам перезапустится,
  если запущен как сервис).
- **В терминале**: `bun run tg update` (= `git pull` + `bun install`), затем перезапусти
  сервис (`systemctl --user restart tg-agent` / перезагрузка launchd / `Ctrl+C` +
  `bun run service`).

Агент **сам периодически проверяет** новую версию (раз в сутки, настраивается
`TG_UPDATE_CHECK_HOURS`) и, если она есть, **присылает в бот предложение** обновиться с
кратким списком изменений из CHANGELOG.

## Команды CLI

```bash
bun run setup [имя]                # мастер настройки пользователя (начни с него)
bun run service                    # запустить сервис (все пользователи) на переднем плане
bun run service --once             # один проход по каждому (для проверки/cron)
bun run tg tenant list             # список пользователей (тенантов)
bun run tg tenant add <имя>        # создать пользователя
bun run tg tenant migrate-legacy <имя>  # перенести старую data/ в tenants/<имя>
bun run tg login <имя>             # вход в Telegram для пользователя
bun run tg install-service         # поставить как фоновый сервис + автозапуск
bun run tg uninstall-service       # удалить фоновый сервис
bun run tg service-help            # логи / стоп / рестарт сервиса
bun run doctor                     # проверить окружение
bun run tg doctor --prepublish     # проверка перед публикацией (нет утечек)
bun run tg version                 # версия + проверка обновлений
bun run tg update                  # обновиться до последней версии
bun run tg permissions list        # кому агент может писать
bun run tg permissions revoke <id> # отозвать разрешение
bun run tg help                    # все команды
```

## Движки (engine matrix)

| Движок | Аутентификация | Непрерывная сессия | Статус |
| --- | --- | --- | --- |
| **Claude** (Claude Code) | локально настроенный Claude Code: `claude setup-token` или интерактивный `claude` login. Если задан `ANTHROPIC_API_KEY` — оплата через API | да (streaming + resume) | основной |
| **Codex** | `codex login` (подписка ChatGPT) или `OPENAI_API_KEY` | да (resume по thread id); без SDK — деградация на CLI `codex exec` | рабочий |

Движок по умолчанию — `claude` (меняется в `data/config.json` → `agent`, или `TG_AGENT`).

**Про подписку и API-ключи (важно).** Этот проект просто запускает локальный движок
(тот же Claude Code / Codex, что и в терминале) и использует ту же аутентификацию, что
и он. Мы **не** предлагаем обход биллинга. Для Claude: без `ANTHROPIC_API_KEY` идёт
обычная локальная авторизация Claude Code; с ключом — оплата через API (`bun run doctor`
это покажет). Если используешь Claude Agent SDK в своём продукте — сверься с
[условиями Anthropic](https://docs.anthropic.com/en/docs/claude-code/sdk).

## Архитектура

**С Telegram работает только сервис** — единственный процесс-владелец сессий. Он
**мультитенантный**: в одном процессе поднимает несколько пользователей, у каждого свой
Telegram-аккаунт, бот, память и свой **RPC-хаб** (127.0.0.1, отдельный порт, bearer-токен).
Изоляция — через контекст тенанта; lock у каждого свой (второй экземпляр не запустится).
Интерактивные агенты (Claude Code/Codex) обращаются к Telegram через этот хаб.

Память и личные данные — в `tenants/<имя>/` (не в git): сессия, правила-оверрайды,
память, handoff, мониторы, конфиг. Базовые правила-промпты — общие для всех в `rules/`
(в git); личные оверрайды по имени файла — в `tenants/<имя>/rules/`. (До перехода на
мультитенант старая единая `data/` продолжает работать как один пользователь.)

## Безопасность

Кратко (подробно — [SECURITY.md](SECURITY.md)):

- Это **личный локальный** инструмент. Агент работает с **полными правами** (как Claude
  Code без ограничений: все инструменты, включая Bash) — запускай его только на своей
  машине, со своими данными.
- В Telegram действует код-гейт: отправка сообщений/файлов посторонним **запрещена в
  коде** без твоего явного разрешения (себе и в управляющий канал — всегда).
- Чувствительные файлы (сессия, конфиг, `.env`) нельзя отправить как файл через агента.
- RPC-хаб на localhost требует bearer-токен; секреты маскируются в логах.
- Личные данные (`tenants/`, `data/`) и заметки (`spec/`) — в `.gitignore`; перед
  публикацией гоняй `bun run tg doctor --prepublish`.
