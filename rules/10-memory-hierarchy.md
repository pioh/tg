# 10 · Иерархия памяти

Вся память — на диске, в понятной структуре. Любой агент любой платформы обязан
её читать и поддерживать. Меняй память **только в `data/`**.

## Карта файлов

| Путь | Что это | Режим |
|------|---------|-------|
| `rules/` | Базовые правила (git, общие) | только чтение |
| `data/rules/` | Личные правила, переопределяют базовые по имени файла | чтение/запись |
| `data/handoff.md` | Актуальный статус «здесь и сейчас» для следующего агента | перезапись |
| `data/progress.txt` | Append-only журнал значимых действий | только дозапись |
| `data/qa/<дата>.md` | Дословные просьбы человека | только дозапись |
| `data/memory/*.md` | Долговременная структурированная память (люди, чаты, факты) | чтение/запись |

## Правила едины для всех платформ

Claude Code, Codex (и будущие движки) используют **одну и ту же** память в `data/`.
Поэтому:

- Не дублируй знания в `CLAUDE.md`/`AGENTS.md` — там только ссылка на эту иерархию.
- Когда обновляешь память/правила, делай это в `data/` через MCP-инструменты
  (`mem_rule_set`, `mem_handoff_set`, `mem_progress_append`, `mem_note_set`,
  `mem_qa_record`) или прямым редактированием файлов в `data/`. Тогда изменения
  увидит следующий агент **любой** платформы.

## Дословные просьбы (data/qa)

Каждую просьбу человека фиксируй **дословно**, без перефразирования, с датой и
источником, в `data/qa/<дата>.md`.
- Команды из Telegram (бот/управляющий канал) сервис пишет в qa автоматически.
- Просьбы из других каналов (CLI, прямой чат с тобой) — пиши через `mem_qa_record`.
QA — это «сырой» исторический след: что именно просили и когда. Его не редактируют.

## Правила храним в ФИНАЛЬНОМ виде

Различай два слоя:
- **QA** хранит историю формулировок (как просили в разные моменты).
- **Правила** (`data/rules/`, `data/memory/`) хранят **итог** — как человек хочет,
  чтобы было **в результате**, с учётом всех уточнений.

Когда человек уточняет/меняет правило — **перепиши соответствующий файл правила
целиком в финальное состояние** (`mem_rule_set`), а не дописывай противоречивые
слои. Старую формулировку искать не нужно — она остаётся в QA. Цель: чтобы файл
правил всегда читался как единственная актуальная инструкция.

## Долговременная память (data/memory)

Сюда складывай устойчивые факты, которые помогают вести чаты: профили людей, тон
общения с конкретными контактами, договорённости, важные ссылки/ID. Одна тема —
один файл (`mem_note_set`). В `data/handoff.md` держи только оперативное.
